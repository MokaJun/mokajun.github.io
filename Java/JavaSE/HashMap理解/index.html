Hash表

数组：数组存储区间是连续的，占用内存严重，故空间复杂度很大，但数组的二分查找时间复杂度很小，为 o（1），数组的特点：查找速度快、插入和删除效率低

链表：链表存储区间离散，占用内存比较宽松，故空间复杂度很小，但时间复杂度很大，为 o（n），链表的特点：查找速度慢、插入和删除效率高

哈希表：综合了数组和链表的特性，不仅查找速度快，插入和删除的效率也很高，哈希表有多种不同的实现方式，面对哈希冲突，哈希表使用最常用的方法——“拉链法”，就是我们理解的“链表的数组”，如下图所示：

HashMap的put方法

要搞清楚 HashMap ，首先要知道 HashMap 是什么，它实际是 Node<K,V>[] table，即哈希桶 数组。

为了方便，我们在这里分出两个概念，hash()和hashcode()，其实他们都是获取哈希值的算法，或者说是映射，但是方法不同，得到的哈希值就不同。

hashCode()是直接映射出hashCode值，

而文中的hash()是HashMap为了让第一次hashCode生成的Node.Key用hash(Node.Key)又进行了高位运算、取模运算 再生成的hash值。

是因为hash能够对hashCode分布更均匀，然后经过一定的处理，作为此元素在这个table[]里的index

至此，put方法完成。

HashMap基本原理

HashMap实际就是Node<K,V>[] table。而这个Node[]的索引就是hash(Node.Key)得到的。

{注：Node也重写了hashcode(key,value),储存在Node节点本身，用来确定在当前Node[]索引下的具体Node}。

举个例子

Map<String,Object> map = new HashMap<>();
map.put<"Str",0>

我们首先通过hashCode(“Str”)计算出这个键值对的Key的hashCode

然后我们通过hash()算法，来定位这个键值对在HashMap即Node[]里的索引位置，找到位置储存

当我们以后想取出时，就根据hash()找到索引位置，然后根据索引处获得的链表，用hashCode()找到对应键值对

哈希冲突：loadFactor（装填因子）

加载因子是表示Hash表中元素的填满的程度（可能是纵向的，也可能是横向的）.若:加载因子越大,填满的元素越多,好处是,空间利用率高了,但:冲突的机会加大了.反之,加载因子越小,填满的元素越少,好处是:冲突的机会减小了,但:空间浪费多了.

冲突的机会越大,则查找的成本越高.反之,查找的成本越小.因而,查找时间就越小.

把哈希桶数组Node[]想象成横向长度。把每个索引对应的Node链表想象成纵向长度。当处于极端情况时，无论横向太长，还是纵向太长，都不利于我们使用。横向太长，占用空间变多。纵向太长，储存和遍历困难，浪费时间。

HashMap中的

threshold = (int)(capacity * loadFactor);

极端的想，

如果loadFactor很小很小，那么map中的table需要不断的扩容，导致空间越来越大，而且每次resize非常浪费性能！

如果loadFactor很大很大，那么当map中table放满了也不要求扩容，导致冲突越来越多，解决冲突而起的链表越来越长！

所以就需要在时间和空间成本之间权衡，其实就是根据实际情况确定哈希桶数组的大小，并在此基础上设计好的 Hash 算法减少 Hash 碰撞，那么通过什么方式来控制map使得 Hash 的碰撞概率低，哈希桶数组（Node[] table）占用较少的内存呢？答案就是好的 Hash 算法和扩容机制。好的Hash算法可以让每个索引被填满，而且存放的比较均衡。扩容机制则会保证数组横向纵向的长度不会失衡。

因此我们引入了负载因子。规定其为0.75。当HashMap中存入的元素个数/长度>0.75时，长度扩容为原来的2倍。一旦长度扩容，每个索引下挂的元素链表/红黑树 也会变化，会使他们重新rehash然后排序。重新排序后，他们的纵向长度就会变短。
