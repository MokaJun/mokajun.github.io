Tomcat部署问题：

HTTP Status 500 – Internal Server Error

Exception

jakarta.servlet.ServletException: 实例化Servlet类[org.springframework.web.servlet.DispatcherServlet]异常

Root Cause

java.lang.NoClassDefFoundError: javax/servlet/http/HttpServlet

血的教训，错误：HTTP Status 500-Internal Server Error，报错内容是无法实例化servlet。原因是Tomcat10中的Servlet-api依赖包与Maven导入的Servlet不兼容所致。

解决方法：

直接降级到9就好了。（真的是脑瘫）

服务器上传/下载时文件名中文乱码

先说结论，不同于消息转换器，我们还需要设置一个过滤器来把request设置成utf-8

在web.xml里设置，注意放在servlet标签前

<filter>
  <filter-name>encoding</filter-name>
  <filter-class>
    org.springframework.web.filter.CharacterEncodingFilter
  </filter-class>
  <init-param>
    <param-name>encoding</param-name>
    <param-value>UTF-8</param-value>
  </init-param>
  <init-param>
    <param-name>forceEncoding</param-name>
    <param-value>true</param-value>
  </init-param>
</filter>
<filter-mapping>
  <filter-name>encoding</filter-name>
  <url-pattern>/*</url-pattern>
</filter-mapping>

通过阅读CommonsMultipartResolver源码

protected String determineEncoding(HttpServletRequest request) {
    String encoding = request.getCharacterEncoding();
    if (encoding == null) {
        encoding = this.getDefaultEncoding();
    }

    return encoding;
}

这种情况原因是CommonsMultipartResolver这个文件解析器的编码方式是从request来get编码格式的。

所以我们可以通过更改request的编码格式，或者强行改变CommonsMultipartResolver的编码格式两种方法来避免中文乱码。后者治标不治本，方法如下：

需要检查
xml 

<bean id="multipartResolver" class="org.springframework.web.multipart.commons.CommonsMultipartResolver" 
p:defaultEncoding="UTF-8" 
p:maxUploadSize="5400000" 
p:uploadTempDir="fileUpload/temp" ></bean>

里面是不是配置了utf-8，因为默认是iso编码（这种情况request.getCharacterEncoding获得值是null,其实就是ISO编码）。
其次需要就检查文件上传的那个jsp页面编码里面的编码。包括pageCode,charset和head标签中meta编码最好都是utf-8编码。其他的想tomcat的 server.xml和项目的web.xml都是可以直接使用默认（有网友指出这两个地方需要修改，亲测这两个我都没改，只是保证了CommonsMultipartResolver和jsp页面编码就ok了）。

数据库报错

在DG中连接数据库时timezone问题

Server returns invalid timezone. Need to set‘serverTimezone’property

使用mybatis连接数据库时抛出异常

错误信息：

Error updating database.

Cause: java.sql.SQLException: The server time zone value ‘ÖÐ¹ú±ê×¼Ê±¼ä’ is unrecognized or represents more than one time zone. You must configure either the server or JDBC driver (via the serverTimezone configuration property) to use a more specifc time zone value if you want to utilize time zone support.

问题原因：mybatis的config.xml配置文件中，设置连接数据库的环境时，没有指定server的timezone

解决方法：在原先配置基础上 加上 &serverTimezone=GMT%2B8

<property name="url" value="jdbc:mysql://localhost:3306/mybatis?
useUnicode=true&amp;characterEncoding=UTF-8&amp;serverTimezone=GMT%2B8"></property>

SSM

org.apache.ibatis.binding.BindingException: Invalid bound statement (not found)

org.apache.ibatis.binding.BindingException: Invalid bound statement (not found): com.southwind.repository.AccountRepository.findAll

此异常的原因是由于mapper接口编译后在同一个目录下没有找到mapper映射文件而出现的。由于maven工程在默认情况下src/main/java目录下的mapper文件是不发布到target目录下的。

在pom中添加build标签 注入依赖即可

注入后记得更新pom。。。血的教训

Caused by: org.apache.ibatis.builder.BuilderException: Wrong namespace.

在这里其实是因为扫描的xml文件中，没有能和动态代理实现类的名称对应上的namespace

1. 首先myBaits根据AccountRepository接口，根据动态代理创建了实现类
    AccountRepository accountRepository = sqlSession.getMapper(AccountRepository.class);
2. 配置MyBatis SqlSessionFactory时，扫描了repository文件夹下的.xml文件，记载了它们的namespace
3. 该实现类会根据方法名来调用同名namespace对应的xml文件的对应方法
4. service组件在使用AccountRepository时，spring根据类型，自动装载了AccountRepository的对应实现类，也就是上面的实现的代理类

Spring Boot

各种错误首先排查application.yml和pom.xml

血的教训，一定要先看一下application.yml的图标对不对，有的时候打错一个字母。。。

如果application没错的话，很多东西是可以自动加载和高亮的。反过来，没自动补全可能就是出了问题

而且pom.xml也有可能加载出问题，可以先重写一下试试。

SpringBoot 发起一次请求却走两次 Controller 和 拦截器的问题

资源路径取不到时，有的设备就会出现两次请求，包括但不限于css、ico等…(就比如没有ico就先把路径设置为#)

林子大了什么鸟都有，这坑壁问题，八竿子打不着原因还不报异常！淦！

而且这个问题非常玄学，有的controller会请求两次，有的不会。。。

而且在发生问题的controller里 有的方法会，有的方法不会，NT，RNM

NT问题又出现了, 放置了favicon.ico但是Spring Boot有时候扫不到，最好手动指定一下

<link rel="shortcut icon" href="../resources/favicon.ico" th:href="@{/static/favicon.ico}"/>

对想测试的方法加@Test注解时 一定要注意导的包是org.junit.jupiter.api.Test

有个名字非常类似的包是junit的，坑比

错误信息：java.lang.NullPointerException

在测试过程中，一定记得加@SpringBootTest注解

因为不加这个的话，单元测试就不知道是在IoC容器中取对象，自然就是空指针了
