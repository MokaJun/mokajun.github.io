秒杀系统优化

2021-07-04- 2022-02-12
665- 2m- -
Database > Redis-Redis
Moka☆君
查看链接

一、最基本的购物系统，下订单时用mysql的update set stock = stock -1

二、增加Redis一级缓存，这一步的思路时利用redis内存中数据进行判断，节约了sql的访问

  1. @PostConstruct利用Spring注解，写一个服务启动初始化的方法，把sql的stock存入redis里

  2. 在下订单时增加一个判断步骤：利用redis存入的stock来做判断，符合条件的继续运行sql语句，不符合的直接返回

    注：stock<0时还原+1，是为了防止少买。请结合try-catch里的还原+1一起食用。

三、增加内存标记。因为在并发访问量远大于redis库存时，频繁的删除和还原redis服务器的stock也是浪费性能的。

  1. 定义一个ConcurrentHashMap来当作内存中的标记，放商品的id和是否售完的Boolean值

  2. 先判断Boolean标记再决定是否执行业务逻辑（文中用包装类的null）。且当商品售空时，改变Boolean标记

  3. 如果商品try更改sql出现异常，同样需要catch中还原

四、业务逻辑服务器的分布式问题：即jvm缓存在分布式集群下的不一致问题

当我们利用内存标记来节省redis服务器的读写开销时，使用的ConcurrentHashMap是基于JVM的，也就是说，多个业务逻辑实例并没有办法互通数据。

问题就是：当(ip+port) 服务器 A执行sql语句发生异常，然后还原map的Boolean值为null，以便后续继续执行秒杀逻辑时。(ip+port) 服务器 B并没有办法更新到这一还原。由于利用map判断是否执行后续逻辑值，所以服务器B就永远不会执行后续。

解决思路：找一个可以同步map的方法

解决方法1：使用zookeeper实时同步集群jvm缓存 || 疑问：为什么不直接用redis做内存标记呢

zookeeper服务端=文件系统+监听通知机制。就是当zookeeper的值一旦更改，它会主动通知那些监听它的应用程序们，并执行自定义的watch方法（在这更改map标记）。所以我们可以储存一个标记在zookeeper上，一旦zk标记改动，就执行watch方法改动map标记。

  1. 首先在stock<0时，在代码块里添加伪码：向zookeeper服务端传递内存标记 商品id 为 true，随后添加zookeeper的监听
  2. 其次在sql异常的catch语句中，同样需要增加伪码：向zookeeper服务端传递内存标记 商品id 为 false（一旦发现改动，执行watch）
